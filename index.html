agentic-kb

AI-native, layered knowledge operations — install, run /kb setup, start working.

"How do I actually work AI-natively — with agents, not just chat?"
"How does my team share knowledge so agents can use it?"
"Is there something I can just install and go?"

If that sounds familiar: agentic-kb gives you a layered knowledge system that agents maintain. One command — /kb — handles capture, evaluation, cross-referencing, and promotion. You think, the agent does the bookkeeping.

v2.0 · 2026-04-18

Why existing approaches fail

Knowledge decay

Wikis accumulate. Pages drift out of sync with reality. Nobody trusts them. Nobody maintains them. Eventually, nobody reads them.

Context loss

Decisions are buried in Slack threads, meeting notes, inbox replies. Who decided what, when, and why becomes unknowable.

Manual curation fatigue

Someone has to triage, tag, cross-link, archive. Doing it well is a full-time job nobody has time for.

Silos everywhere

Personal notes, team workspaces, org strategy, company OKRs — each in its own tool. Nothing flows up or down coherently, and agents can't reach any of it.

The solution — five layers, one command

Content is organized into five layers. Only the personal layer is required. Higher layers are optional and declared in the user's config. One command — /kb — routes everything.

The evaluation gate — five questions

Applied at every persistence boundary: capture, digest, promote, publish. This is the system's immune system.

  1. Does this strengthen a position?  (will a topic file change?)
  2. Does this inform a decision?  (does it provide evidence for an open decision?)
  3. Would you reference this again?  (durable signal vs. passing observation?)
  4. Is this actionable?  (does it imply a concrete next step?)
  5. Does this already exist?  (avoid duplicates)

Scoring: 0/5 → discard (logged with reason). 1–2/5 → finding only (offer idea creation if novelty detected). 3+/5 → finding + topic update + possibly a new decision or idea.

The gate is never silent. Every accept and every reject is logged with the rationale. If the user insists, the agent will persist with a low-confidence flag — but it always explains why.

Four kinds of memory

Each kind has a distinct purpose and context cost. Splitting them by access pattern is what keeps the agent fast.

One command

There is exactly one user-facing command: /kb. The agent infers the layer and action from context.

/kb                        → status (if bare) or auto-detect
/kb [text/URL/path]        → capture + evaluate
/kb review                 → process pending items in inputs/
/kb promote [file]         → push to team KB
/kb digest team            → pull team changes into personal findings
/kb idea [text]            → create an idea (seed)
/kb develop [idea]         → sparring session on an idea
/kb decide [description]   → open a new decision
/kb task done [item]       → complete a task
/kb start-day              → morning briefing
/kb end-week               → Friday 15:00 weekly summary
/kb present [topic]        → versioned HTML presentation (light + dark)
/kb setup                  → run the interactive onboarding wizard

Decisions as first-class objects

Strategic and tactical choices live in their own files with explicit lifecycle, stakeholders, and evidence trails — not buried in prose. When a decision resolves, the agent archives it, updates affected topics with a changelog entry, and closes related tasks.

gathering-evidence → under-discussion → proposed → decided → archive

Ideas as incubation objects

Observations with novelty value that aren't ready to shape a topic become ideas — tracked with their own lifecycle, developed via sparring (/kb develop), and promoted when mature. The agent plays devil's advocate, surfaces assumptions, and tracks the reasoning chain.

seed → growing → ready → shipped | archived

Vision, mission & goals

Each layer can declare a vision (north star), mission (what we do), and goals (measurable targets). These actively steer the system — the evaluation gate checks strategic alignment, tasks are prioritized by goal relevance, and cross-layer promotions are grounded in mission fit.

Presentation-ready output

/kb present [topic] generates a self-contained HTML artifact with both light and dark themes, an in-page toggle, a subtle v{version} · {date} watermark on the intro slide, and a changelog appendix. This very page is an example.

One source, three harnesses

Skills and agents are cross-agent portable. One set of sources feeds VS Code Copilot Chat, Claude Code, and OpenCode via a generated per-harness manifest.

Install

Claude Code

/plugin marketplace add https://github.com/<org>/agentic-kb
/plugin install kb@agentic-kb

VS Code Copilot Chat

// settings.json
{
  "chat.plugins.marketplaces": [
    "<org>/agentic-kb"
  ]
}

OpenCode

git clone https://github.com/<org>/agentic-kb
cd agentic-kb
scripts/install --target opencode --global

Seven core principles

  1. Lean by default — every piece of content must earn its place.
  2. Signal over noise — relevance is measured against themes, not "interesting".
  3. Decisions as first-class objects — tracked explicitly, not buried in prose.
  4. Stakeholders inform action — the agent suggests who to involve.
  5. Always offer next steps — never leave the user staring at output.
  6. Multiple workstreams — one personal KB can track parallel tracks.
  7. Presentation-ready — any topic can become a versioned HTML artifact.
